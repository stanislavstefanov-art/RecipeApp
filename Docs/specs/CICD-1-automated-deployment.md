# CICD-1 — Automated Deployment

## Problem

Deploying RecipesApp to Azure currently requires running ~12 sequential `az` commands by
hand (resource group, Bicep deployment, 7×`az webapp config appsettings set`, backend zip
deploy, two SWA deploys). Re-running any of them after a code change is a manual chore,
and the deploy guide drifts from reality every time something changes. CI exists for
build/test only — nothing deploys.

The deployment guide also asks the operator to store an `AZURE_CREDENTIALS` service
principal secret in GitHub, which has to be rotated periodically and is the kind of thing
that gets pasted into Slack and forgotten about.

## Goal

After a one-time bootstrap (≤ 60 seconds), every push to `main` deploys the right thing
to Azure — infra, backend, React, Angular — without any manual `az` commands. PRs get
preview URLs. No long-lived secrets in GitHub. Bicep is the source of truth for all
infrastructure and app settings.

---

## Design summary

### One-time bootstrap

A single script `infra/bootstrap.ps1` (with a `.sh` companion) creates everything that
*can't* be created from inside a workflow on a fresh Azure subscription:

1. Resource group
2. Azure AD application + service principal
3. GitHub OIDC federated credentials (one per trigger: `main`, `pull_request`)
4. RBAC: `Contributor` on the resource group for the SP
5. JWT signing key — generated once, stashed in a temp file, fed into the first Bicep run

The script prints the three values needed as GitHub repository **variables** (not
secrets, since they are not sensitive on their own without the OIDC trust):

- `AZURE_CLIENT_ID`
- `AZURE_TENANT_ID`
- `AZURE_SUBSCRIPTION_ID`

After the bootstrap, the operator never runs `az` again.

### OIDC, not secrets

Workflows authenticate via GitHub's OIDC provider:

```yaml
permissions:
  id-token: write       # OIDC token request
  contents: read

steps:
  - uses: azure/login@v2
    with:
      client-id: ${{ vars.AZURE_CLIENT_ID }}
      tenant-id: ${{ vars.AZURE_TENANT_ID }}
      subscription-id: ${{ vars.AZURE_SUBSCRIPTION_ID }}
```

The federated credential is scoped to a specific repo + trigger, so the trust is narrow:
only this repo's `main` push (or PR) can assume the role.

### Bicep owns all configuration

`infra/modules/app-service.bicep` grows from 4 settings to ~15. Every `*:Provider`
switch, JWT settings, database provider — all in Bicep. Secrets (Anthropic key, JWT
signing key, SQL connection string) live in Key Vault and are referenced via
`@Microsoft.KeyVault(...)`.

The 7 `az webapp config appsettings set` lines disappear from the deploy guide.

### Per-PR previews

Static Web Apps' built-in `Azure/static-web-apps-deploy@v1` action handles preview
environments natively: pass the PR event, get a `https://<random>.azurestaticapps.net`
URL automatically posted as a PR comment, torn down on close. No extra infra.

### `what-if` PR check on infra changes

Pull requests touching `infra/**` get an `az deployment group what-if` PR check that
posts the resource diff as a comment, the same way the existing `claude-review` workflow
posts review comments. Catches Bicep mistakes before merge.

## Workflows after this change

| Workflow | Trigger | Purpose |
|---|---|---|
| `infra-deploy.yml` (new) | push to `main` on `infra/**`, manual dispatch | `az deployment group create` |
| `infra-validate.yml` (new) | PR on `infra/**` | `az deployment group what-if`, posts diff comment |
| `backend-ci.yml` (extended) | push to `main`, PR | build + test + (on `main`) deploy |
| `frontend-react-ci.yml` (extended) | push to `main`, PR | lint + build + deploy (production on `main`, preview on PR) |
| `frontend-angular-ci.yml` (extended) | push to `main`, PR | lint + build + deploy (production on `main`, preview on PR) |

## Bicep changes

### `infra/main.bicep`

Add one `@secure()` parameter:

```bicep
@secure()
@description('JWT signing key — generated by bootstrap script, stored in Key Vault.')
param jwtSigningKey string
```

Pass it to the Key Vault module.

### `infra/modules/key-vault.bicep`

Add a third secret:

```bicep
resource secretJwtSigningKey 'Microsoft.KeyVault/vaults/secrets@2023-07-01' = {
  parent: keyVault
  name: 'Jwt--SigningKey'
  properties: {
    value: jwtSigningKey
  }
}
```

### `infra/modules/app-service.bicep`

Expand `appSettings:` to include every value currently set manually in the deploy guide:

```bicep
appSettings: [
  // existing
  { name: 'APPLICATIONINSIGHTS_CONNECTION_STRING', value: appInsightsConnectionString }
  { name: 'Claude__ApiKey', value: '@Microsoft.KeyVault(VaultName=${keyVaultName};SecretName=ANTHROPIC-API-KEY)' }
  { name: 'ConnectionStrings__RecipesDb', value: '@Microsoft.KeyVault(VaultName=${keyVaultName};SecretName=ConnectionStrings--RecipesDb)' }
  { name: 'ASPNETCORE_ENVIRONMENT', value: 'Production' }
```

## GitHub repository configuration

After the bootstrap script runs, the operator sets these once:

**Variables** (not secrets):
- `AZURE_CLIENT_ID`
- `AZURE_TENANT_ID`
- `AZURE_SUBSCRIPTION_ID`
- `AZURE_RESOURCE_GROUP` (e.g. `RecipesApp`)
- `AZURE_PREFIX` (e.g. `recipes-prod-abc123`)
- `API_HOSTNAME` (e.g. `recipes-prod-abc123-api.azurewebsites.net`) — populated after first infra deploy

**Secrets**:
- `SQL_ADMIN_PASSWORD`
- `ANTHROPIC_API_KEY`
- `JWT_SIGNING_KEY`

The deployment tokens for Static Web Apps are pulled from Key Vault at deploy time
(stored there by the bootstrap or first infra deploy), so they are not GitHub secrets.

## Out of scope

- Deployment slots (blue/green, staging swap) — F1 free tier doesn't support them, and
  this is a learning project.
- Multi-environment deploys (separate dev / staging / prod resource groups). Single prod
  for now.
- Self-hosted GitHub runners.
- Approval gates / `environment:` protection rules. Could be added later by gating the
  deploy jobs on a `production` environment with required reviewers.
- Automated dependency updates (Renovate / Dependabot) — covered separately if wanted.
- Release-please / tag-triggered deploys. Push-to-main is sufficient for now.
- Rolling back from a previous artefact (the Bicep is idempotent so re-running an old
  workflow run does the right thing; explicit rollback tooling is a future concern).

## Acceptance criteria

1. A fresh operator runs `pwsh infra/bootstrap.ps1` once, sets six repo variables and
   three secrets, and after that never runs `az` again.
2. Pushing a change under `infra/**` to `main` triggers `infra-deploy.yml`, which runs
   `az deployment group create` and finishes green.
3. Pushing a change under `Backend/**` to `main` triggers `backend-ci.yml`, runs unit +
   integration tests, then deploys to App Service. `curl https://<api>/health` returns
   `Healthy` after the workflow finishes.
4. Pushing a change under `Frontend/**` to `main` deploys to the React Static Web App
   with the correct `VITE_API_BASE_URL` baked in.
5. Pushing a change under `FrontendAngular/**` to `main` deploys to the Angular Static
   Web App with the correct production `apiBaseUrl`.
6. Opening a PR that touches `Frontend/**` or `FrontendAngular/**` results in a comment
   from the SWA action with a preview URL. Closing the PR tears the preview down.
7. Opening a PR that touches `infra/**` results in a `what-if` diff comment showing what
   resources would be created / modified / deleted.
8. No `AZURE_CREDENTIALS`-style long-lived service principal secret exists in the repo's
   GitHub secrets.
9. `Docs/deployment/azure.md` collapses from 7 manual steps to: "run `bootstrap.ps1`,
   set repo variables, push to `main`."
